Executive Summary

This report refers to most modules in the Finance application, plus the start screen.

Note: Due to the high number of issues found and the repetitive nature of them, not all issues have been logged in this report. We trust that the development team, that knows the product and the code base, can deduce where else fixes could be applied.

Observations

  • Older coding practices were observed, CSS styling is applied on top to make the application seem more modern. HTML best practices aren't always applied
  • A basic level of accessibility is not achieved:
    • Critical issues: Not all functionality is accessible via keyboard, screen reader or speech recognition. Examples: buttons that cannot be activated because they don't receive focus, form controls that cannot be filled in because they have no programmatically associated labels to describe their purpose.

Legal notes

According to the European Accessibility Act, legacy applications can fall into 2 categories:

  • applications that don't receive any updates after July 28 2025 and have until 2030 to comply with the law
  • applications that receive updates after July 28th 2025 and must comply with the law

Conclusions

  • Carefully consider whether the application should receive updates or not after July 2025, as this will decide the amount of work needed and the deadline for complying with the law.
  • We recommend a rewrite of the application, preferably using the Windmill Design System (with its accessible components and its documentation on accessibility).
  • Perform periodic automated accessibility testing, using browser tools (e.g. WAVE, ARC Toolkit, Axe Dev Tools or Access Assistant). Keep in mind that automated testing cannot cover all accessibility issues!

Issue: Redundant title text. Title attribute text is the same as text or alternative text.

Fix: If a text label for a form control is visible, use the <label> element to associate it with its respective form control. If there is no visible label, either provide an associated label, add a descriptive title attribute to the form control, or reference the label(s) using aria-labelledby. Labels are not required for image, submit, reset, button, or hidden form controls.

Issue: Most inputs and labels aren't associated. Some good practices were found in radios and checkboxes.

Screen reader users won't know what kind of data to fill in if they don't know what the input represents. Speech recognition users will also not be able to speak the visible label to access the inputs, because of the lack of association between inputs and visible labels.

Fix: Associate inputs with labels. Use label elements and the for attribute to reference inputs by ids.

2.1.1: Keyboard
Entire sample

Result: Failed
Entire sample

Observations:

Factuurafhandeling

Issue: Device dependent event handler. An event handler is present that may not be accessible.

Fix: Ensure that critical functionality and content is accessible by using a device independent event handler (which responds to both keyboard and mouse) or by using both a mouse dependent and a keyboard dependent event handler.

Issue: The functionality provided in the header of the app (e.g. home, help, logout) cannot be accessed by keyboard users or screen reader users.

Fix: Use the button element instead of spans with background images. Use a select element for the dropdown in the left corner.

Issue: Keyboard users and screen reader users can't use the Space key to check/uncheck a checkbox in a table.

Fix: Investigate why the checked state isn't retained on Space key interaction.

Issue: Clicking the button in the table to view an item sometimes shows an error dialog. The keyboard focus remains in the table, underneath the error dialog. Keyboard users cannot reach or close that dialog.

Fix: When a dialog appears, the user's focus should be switched to the dialog (or an element inside the dialog such as the heading or the close button). Pressing the Escape key should close the dialog if an action from the user isn't required to close it.

After the user closes the dialog via keyboard, the keyboard focus should return to the element that triggered the dialog.

Issue: All elements that can be activated with mouse click must also receive focus and Space/Enter interaction via keyboard (this depends on the element type).

The tabs component that shows additional content on hover doesn't work for keyboard users and screen reader users.

Fix: Implement an accessible tooltip behavior:

  • The tooltip must be shown not only on mouse hover, but also on keyboard focus
  • The tooltip must be dismissible via user interaction (Tabbing to a different element or pressing the Escape key)

The tabs should also be implemented so that they're correctly announced to screen reader users. Tabs Pattern | APG | WAI | W3C.

Issue: The keyboard focus isn't maintained when performing actions in the app.

Example: In "Factuurafhandeling", choosing to sort a column via keyboard moves the keyboard focus a few stops back, in a form input, instead of remaining on the sort trigger inside the table header.

Fix: Maintain focus on the element that triggers the column sort.

Warning: In "Factuurafhandeling", when opening the "Selecteren Kredietbeheerders" popup, Keyboard users and screen reader users might be confused by the fact that focus can move behind the popup, in the links in the header.

Fix: Implement focus trapping for the popup.

Issue: The button titled "Selecteren kredietbeheerder" cannot be activated by keyboard users via Enter key.

Fix: Investigate issue for input type image or use a button element with click events. Click events handle keyboard events too by default.

Issue: In "Aanvraagformulieren", "Inkoop" and similar modules, the left side menu cannot be accessed by keyboard users and screen reader users. The menu toggle also cannot be accessed by these users.

Fix: Do not use div elements for interactive purposes. Use buttons for actions (e.g. revealing/hiding nested menu items) and links for navigating to new pages.

Issue: In "Kredietbewaking", when opening the "Gegevens filteren op basis van ingestelde filterwaarde..." popup via keyboard, the focus remains underneath the popup. Keyboard users and screen reader users don't have access to this popup.

Fix: Implement focus management for keyboard and screen reader users. When a popup appears that users need to get to in order to complete a flow, move the focus to that popup. After the popup is closed, restore the focus to the popup trigger (the button that opened it).

Issue: In the "ProjectBeheersing" module, the search button isn't reachable by keyboard.

Fix: Remove tabindex="-1" from the search element (input type image).

2.4.3: Focus Order
Entire sample

Result: Failed
Entire sample

Observations:

Factuurafhandeling

Issue: Tabindex. A positive tabindex value is present.

Fix: If the natural tab order is already logical, remove the tabindex. Otherwise, consider restructuring the page so that tabindex is not needed. If tabindex is maintained, ensure that the resulting navigation is logical and complete.

Issue: Focus order doesn't always match the visually expected order. This can cause confusion, especially for sighted keyboard users.

Fix: Do not use positive values for tabindex if there's no legitimate reason for it. Positive tabindexes change the order in which elements receive focus. When this order is badly managed (e.g. forgetting to add a tabindex or adding the wrong number value) users will experience elements getting focused in the wrong order compared to the visual order.

Issue: In "BewakingInvordering", "Brieven" page, after going to one of the items in the table, the keyboard focus is out of order.

Fix: Remove tabindex values. Do not use positive tabindex values.

Note:

  • Avoid adding tabindex="0" to already interactive elements (button, input, link/anchor etc.).

  • Do not use a negative value for tabindex without supplying the JavaScript code that eventually turns that negative tabindex to a zero value. Negative tabindex prevents keyboard users from focusing elements.

  • Do not use positive values for tabindex without a legitimate reason. Positive tabindexes, when used improperly, change the order in which elements receive focus, causing confusion for users.

Warning: Filling in a form is difficult with the current focus order. Users have to go through an entire form, reach the end of it, and then scroll back up to find a Save or Submit button. This is difficult especially for keyboard users, screen reader users and mobile users.

Fix: Add submit button after forms (in the DOM order).

4.1.2: Name, Role, Value
Entire sample

Result: Failed
Entire sample

Observations:

Issue: Form controls don't have an accessible name and they're not associated with the visible labels. This applies to most inputs in the entire application (in forms, in tables). Sometimes span elements are used for labels instead of the label element.

Screen reader users don't know what the form control is for or what data to fill in. Voice recognition users can't use the visible label name to access these form controls because the labels and form controls aren't connected in code.

Fix: Provide accessible names to form controls. Associate labels and form controls together. The best way to do that is by using the label element together with the for attribute.

Warning: Buttons (or inputs with type="image" in this case) have accessible names given by titles, but these are not accessible to keyboard users and can only be used by sighted mouse users.

Fix: If they can be converted to other elements, consider using the button element instead, and add accessible names via aria-label. Use keyboard accessible tooltips to give users hints about the purpose of the icons.

Issue: The wrong element (link) is used to provide table sorting functionality. Screen reader users will assume this element will navigate them to a new section or page. These elements will show up in screen reader user's list of links.

More info: Using javascript: URLs as href targets

Fix:

  • button: Buttons usually do actions. Exception: Submit buttons/inputs on forms also navigate.
  • a: Links usually navigate to a new page or a section of the same page.

Use a button element for sorting actions.

Note: Also provide hints that clicking on the column will perform sorting actions:

  • for sighted users, some arrow icons would be useful
  • for screen reader users, aria-sort should be used to communicate the sort status. ARIA: aria-sort attribute

Issue: In "Factuurafhandeling", when opening the "Selecteren Kredietbeheerders" popup, screen reader users don't know that the popup is a dialog and what the title of the dialog is.

Fix: Avoid using table markup for layout. Use a dialog element or the dialog role to provide semantics/meaning to screen readers. Make sure you give the dialog an accessible name: e.g. use aria-labelledby and reference the title of the dialog. Modal Dialog Example | APG | WAI | W3C.

Issue: Required states can't be programmatically determined. Screen reader users are not announced that certain inputs are required. The asterisk used in the "label" is not clear/descriptive on its own and the label also isn't programmatically associated with the input.

Fix: If the asterisk character must be kept, consider adding an explanation for the asterisk notation such as a disclaimer at the beginning of forms that says: "Fields marked with asterisk are required". This message is useful to sighted users but could be hidden from screen reader users if each field is marked as required in code. Use required and aria-required attributes on inputs to signal to screen readers that the input is indeed required.
4.1.3: Status Messages
Entire sample

Result: Failed
Entire sample

Observations:

Issue: The error dialogs aren't announced to screen reader users.

Fix: If the user must interact with the dialog, move the focus into the dialog (on the dialog element, on the title which should be a heading).

The dialog should also receive a proper role (role="dialog") and also an accessible name. Adding an accessible name could be done using aria-labelledby by referencing the heading containing the title of the dialog.

Issue: In "Factuurafhandeling", when searching and after pressing Enter, the loading state isn't announced to screen reader users.

Fix: Announce loading states and UI changes that screen reader users can't be aware of due to not being able to see them. Use aria-live or role="status" to provide updates to screen readers.
